# 투표 기능 성능 이슈 로드맵

> 목표: 투표 리스트/상세 페이지가 **빨리 뜨게 만들고**,  
> 어디를 고치면 얼마나 좋아지는지 **실험하면서 배우기**.

아래 단계들은 **순서대로 하나씩 적용 → 체감 속도 체크**를 목표로 합니다.  
각 단계 설명은 **초등학생도 이해할 수 있게** 최대한 쉽게 적었습니다.

---

## 1단계 – 지금 어디서 느린지 눈으로 확인하기

### 1-1. 브라우저 개발자 도구로 체크

- **무엇을 하나요?**
  - 크롬에서 투표 페이지 열기 → `F12` → `Network` 탭 열기
  - `/predict/[market]` 상세 페이지 들어갈 때:
    - `Document` / `api/sentiment/poll` / `api/sentiment/polls` 요청이
    - **얼마나 오래 걸리는지(시간)** 확인

- **왜 하나요? (초등학생 설명)**
  - “서버에서 **밥 짓느라 느리냐**” vs “브라우저가 **밥 담아서 내놓느라 느리냐**”를 나누기 위해

- **체크 포인트**
  - 대략적인 감으로
    - `/predict/...` 응답이 몇 ms/초 나오는지
    - 주요 API들이 100ms 수준인지, 500ms~1초 이상인지 메모해두기

## 2단계 – 투표 상세 페이지: “너무 큰 레고”를 쪼개기

**파일**: `src/app/predict/[market]/page.tsx`

지금은 이 파일 하나에 **페이지 전체 UI + 상태 + 로직**이 다 들어 있음.

### 2-1. 컴포넌트 쪼개기 (구조 정리부터)

- **무엇을 하나요?**
  - `PredictMarketPage`를 여러 작은 컴포넌트로 나눈다. 예)
    - `MarketHeader` – 아이콘, 제목, 마감까지 남은 시간
    - `MarketStatsCards` – 목표가 / 현재가 / 마감까지 카드
    - `MarketChartSection` – BtcChart / KospiChart / TradingViewChart 선택
    - `TodayResultsSection` – 오늘 결과 동그라미 리스트
    - `BettingPanel` – 잔액/버튼/금액입력/다이얼로그
    - `BettingSummary` – 롱/숏 인원+코인 합계
    - `RulesTabs` – Rules / Market Context 탭
    - `RelatedMarketsGrid` – 다른 시장 링크

- **왜 하나요? (초등학생 설명)**
  - 지금은 **엄청 큰 레고 한 덩어리**라서, 조금만 바뀌어도 **전체를 다시 들어올려야 함** → 힘듦 → 느려짐.
  - 레고를 **여러 작은 블럭**으로 나누면
    - 바뀐 블럭만 다시 움직이면 됨 → 덜 힘들다 → 더 빠르다.

- **완료 기준**
  - UI/동작은 그대로인데, 코드가 여러 컴포넌트로 나뉘어 있고
  - `page.tsx`는 “조립만 하는 얇은 컴포넌트”에 가까워진 상태.

## 3단계 – 투표 상세 페이지를 Server Component 기반으로 바꾸기

지금 `page.tsx` 맨 위에 `"use client";`가 있어서, **전체가 클라이언트 컴포넌트**로 동작 중.

### 3-1. 데이터를 서버에서 미리 준비해서 내려주기

- **무엇을 하나요?**
  1. `page.tsx`에서 `"use client";` 제거 → 기본 Server Component로 전환.
  2. 서버에서 필요한 데이터를 한 번에 가져오기:
     - `poll` 데이터  
     - `user` (닉네임, `voting_coin_balance`)  
     - 코인 시장이면, 오늘 정산 결과(`today-results`)
  3. 가져온 데이터를 props로 쪼갠 클라이언트 컴포넌트에 전달.

- **왜 하나요? (초등학생 설명)**
  - 지금: 손님(브라우저)이 와서 **마트에 가서 재료 사고, 요리까지 직접 함** → 시간 오래 걸림.
  - 바꾸기: 주방(서버)이 **손님 오기 전에 재료 준비하고, 어느 정도 요리까지 해서** 접시에 담아둠.
  - 손님 입장에서는 **“앉자마자 밥이 거의 나와 있는 느낌”** → 빠르게 느껴진다.

- **완료 기준**
  - `/predict/[market]`을 새로고침했을 때:
    - 예전보다 “빈 화면” 시간이 줄고,
    - HTML 안에 이미 데이터가 채워진 상태로 내려오는 것 확인.

## 4단계 – polling(주기적 새로고침) 줄이기

지금은 상세 페이지에서 여러 `useEffect`가 **짧은 간격으로 계속 API를 호출**한다.

### 4-1. 투표 상세 페이지 polling 간격 완화

**대상**: `src/app/predict/[market]/page.tsx` 내부 `useEffect` 들

- **무엇을 하나요?**
  - 인원/코인 합계 polling:
    - 지금: 3초마다 `/api/sentiment/poll`
    - 변경: 5초 ~ 10초 간격으로 늘리기.
  - 정산 상태 polling:
    - 지금: 3초마다 `/api/sentiment/poll`
    - 변경: 5초 ~ 10초 간격, 혹은 “마감 이후 한동안만” polling.
  - Binance 현재가:
    - 지금: 15초마다 REST 호출
    - 변경: 30초 ~ 60초 간격으로 늘려보기.

- **왜 하나요? (초등학생 설명)**
  - 3초마다 계속 “선생님, 이거요?” 하고 물어보면  
    → 선생님도 힘들고, 질문하는 나도 힘듦.
  - 10초마다, 30초마다 물어봐도 크게 문제 없으면  
    → 둘 다 훨씬 덜 힘들고, 느려지는 느낌도 줄어든다.

- **완료 기준**
  - polling 간격을 늘린 뒤:
    - Network 탭에서 초당 요청 수가 줄어들고,
    - 브라우저가 덜 버벅이는 느낌이 나는지 확인.

### 4-2. 홈 투표 섹션 `/api/sentiment/polls` polling 완화

**파일**: `src/components/home/VotingSection.tsx`

- **무엇을 하나요?**
  - 현재:
    - 마운트 시 `fetchPolls()` 1회
    - `setInterval(fetchPolls, 5000)` → 5초마다 전체 시장 폴 데이터 조회
  - 변경:
    - 5초 → 10초 등으로 간격 늘리기.

- **왜 하나요?**
  - `/api/sentiment/polls`는 **여러 시장 정보를 한 번에** 가져오는 무거운 API.
  - 너무 자주 호출하면, 서버/브라우저 모두 피곤해진다.

- **완료 기준**
  - 홈을 켜두고 Network 탭을 보면:
    - 10초에 한 번 정도만 `/api/sentiment/polls`가 날아가는지 확인.

## 5단계 – 차트/실시간 기능은 “천천히 로딩해도 괜찮게” 유지

### 5-1. 차트 dynamic import 상태 확인

**파일**: `src/app/predict/[market]/page.tsx`

- 이미 다음과 같은 형태로 잘 되어 있음:

  ```ts
  const BtcChart = dynamic(
    () => import("@/components/predict/BtcChart").then((m) => ({ default: m.BtcChart })),
    {
      ssr: false,
      loading: () => <로딩 스피너>,
    }
  );
  ```

- **무엇을 하나요?**
  - 이 구조는 유지하되,
  - “차트 이외의 정보들(헤더, 목표가, 잔액, 배팅 카드, Rules 등)”이 **서버에서 미리 준비된 상태**가 되도록 앞 단계들을 먼저 적용.

- **왜 하나요? (초등학생 설명)**
  - “차트”는 무거운 그림이라, **조금 늦게 떠도 괜찮은 영역**.
  - 대신, 차트 옆/위에 있는 글자·버튼·숫자들은 **빨리 보이는 게 중요**하니까
    → 그 부분을 먼저 빠르게 보여주는 작업을 우선한다.

## 6단계 – (선택) 백엔드 사전 계산/캐싱 도입

이 단계는 앞의 구조/렌더링 최적화가 끝난 뒤에 고려.

- **아이디어**
  - `settlement-service`, `btc-klines` 같은 무거운 계산이나 외부 API 호출:
    - 페이지 요청 때마다 하지 말고,
    - **크론/배치 작업에서 미리 계산해서 DB에 저장**.
  - 페이지는 이 “미리 계산된 스냅샷”을 단순 SELECT 해서 사용.

- **왜 하나요? (초등학생 설명)**
  - 손님이 올 때마다 **그때그때 국밥을 처음부터 끓이면** 오래 걸린다.
  - 국물을 미리 큰 냄비에 끓여두고, 손님 오면 **데우기만** 하면 훨씬 빠르다.

- **연결 문서**
  - `docs/xrp-cron-roadmap.md` 등 기존 크론 설계 문서를 같이 참고.
